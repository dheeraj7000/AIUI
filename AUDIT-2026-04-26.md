## Critical (P0) — affect first-run experience

### C-1. Seeded style packs and component recipes are invisible to every user

- **Location:** `packages/design-core/src/operations/style-packs.ts:111` (and the equivalent in `component-recipes`)
- **Symptom:** Brand-new signup → `/style-packs` and `/components` are empty in the dashboard
- **Root cause:** All 6 seeded packs and 57 seeded recipes have `organization_id = NULL` (system defaults). The query is `eq(stylePacks.organizationId, organizationId)`, which excludes NULL rows.
- **Fix:** `or(eq(stylePacks.organizationId, organizationId), isNull(stylePacks.organizationId))`. Same pattern in `listComponentRecipes`.
- **Evidence:** Direct SQL count is 6/57 with all NULL. API call as authed user returns `{data: [], total: 0}`.

### C-2. Multi-tenant contracts remain in API surface despite scope cut

- **Location:** Scope cut announced in CLAUDE.md ("Multi-tenant organizations ... removed"). Reality:
  - DB still has `organizations`, `organization_members`, `invitations`, `tags`, `resource_tags`, `pack_registry`, `pack_ratings`, `usage_events`, `credit_ledger` — 8 tables that no longer have schema definitions in `packages/design-core/src/db/schema/`
  - APIs `/api/projects`, `/api/style-packs`, `/api/component-recipes`, `/api/design-profiles` all require `organizationId` query param
  - Signup creates a hidden "qa-…'s workspace" organization to make this work
- **Risk:** When a future migration drops these tables, all the API queries will break.
- **Fix:** Either (a) drop the API requirement and infer org from the user's single workspace server-side, or (b) write a migration that drops the 8 stale tables and update the API to skip orgId entirely.

## High (P1)

### H-1. MCP tools registered twice on stdio boot

- **Location:** `apps/mcp-server/dist/index.js`
- **Symptom:** Logs show every tool registered twice with separate timestamps:
  ```
  Registered tool: get_project_context (16:286)
  ...
  Starting in local stdio mode
  Registered tool: get_project_context (16:314)   ← second time
  ...
  Starting AIUI MCP server
  ```
- **Impact:** Wastes work on startup; no functional break visible on listed tools, but it's a smell. Likely the bootstrap wires both the unified registry and the per-mode registrar.

### H-2. Stale test asserts removed tool `resolve_tag`

- **Location:** `apps/mcp-server/src/__tests__/tool-registration-smoke.test.ts:30`
- **Symptom:** `resolve_tag` listed in `EXPECTED_TOOLS` but the tag system was cut. Test fails 2/2.
- **Fix:** Remove `resolve_tag` from `EXPECTED_TOOLS`. One line.

### H-3. 404 / not-found pages still use legacy zinc/indigo theme

- **Location:** `apps/web/src/app/not-found.tsx`, `apps/web/src/app/(dashboard)/not-found.tsx`
- **Symptom:** When a public visitor (or my QA test) hits a nonexistent URL, they see a black `bg-zinc-950 / text-white / bg-indigo-500` screen — total visual breakage of the editorial brand.
- **Note:** This was missed by the dashboard migration because `not-found.tsx` doesn't render inside `AppShell`/`editorial-app`.

## Medium (P2)

### M-1. Public visitors get redirected to /sign-in for nonexistent pages

- **Location:** `apps/web/src/middleware.ts`
- **Symptom:** `GET /no-such-page` → 307 → `/sign-in?redirect=/no-such-page`. A public visitor never sees a 404.
- **Fix:** Move the redirect-to-sign-in _after_ an existence check, or whitelist-then-fallthrough so public 404s render.

### M-2. MCP tool argument naming inconsistent (slug vs projectId)

- **Symptom:** `get_project_context` takes `slug`, but `get_theme_tokens` takes `projectId` (UUID). LLMs and humans both have to context-switch.
- **Fix:** Accept either on both. Or pick one — `slug` is more LLM-friendly, `projectId` is more API-friendly.

### M-3. Next.js 16 deprecation warning: `middleware.ts` → `proxy.ts`

- **Symptom:** Compile log: `The "middleware" file convention is deprecated. Please use "proxy" instead.`
- **Fix:** Rename file. Functionality unchanged.

### M-4. `JWT_SECRET` was empty in `.env.local` — app would not have started without intervention

- **Action taken during audit:** Generated a secret with `openssl rand -base64 32` and appended it.
- **Recommendation:** Add a check on startup that prints a friendly error and aborts when `JWT_SECRET` is unset, rather than relying on the runtime stack trace.
